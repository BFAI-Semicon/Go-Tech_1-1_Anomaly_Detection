# 工程横断ゼロショット欠陥検出：Promptable Patch Retrieval（概要書）

## 1. 背景と目的
半導体製造における欠陥検出は、工程・材料・撮像条件の差（Si 半導体／化合物半導体、光学／SEM 等）により、教師あり学習モデルの汎化が難しい。加えて、欠陥の出現頻度が低く、ラベル付けコストが高いという課題がある。

**自己教師あり学習（Self-Supervised Learning, SSL）**は**人手によるラベル付けを一切必要とせず**、画像そのものに内在する構造（領域間の整合性、欠損部の再構成可能性、ビュー変換に対する不変性など）を**モデルが自動生成する予測タスク**として学習させる枠組みである。近年、自然画像領域では SSL によって、ImageNet などの教師あり事前学習に匹敵する、あるいはそれを上回る汎用視覚表現が得られることが示されている。代表的な方式は、(i) 同一画像から作った 2 つのビューの埋め込みを一致させる**自己蒸留型 SSL（DINO／DINOv2 等）**、(ii) 入力パッチの一部をマスクして元画素を復元させる**マスク再構成型 SSL（MAE 等）**である。**欠陥が稀でラベル付けコストが高い半導体製造の文脈において、ラベルを要さずに大量データから汎用表現を獲得できる SSL は、欠陥検出パイプラインの起点として極めて適しており、本研究はその SSL 表現を案件データに対して重み更新せずそのまま用いる。**

本研究開発の目的は、**工程・材料・撮像条件が異なる場合でも汎用的に通用する欠陥検出モデル**を構築することであり、その実現手段として、特徴抽出器に **Vision Transformer（ViT）**を採用する。ViT は **Self-attention によって画像全体のパッチ間関係を一度に集約する**ため、CNN の局所受容野では捉えにくい**スケール変化や撮像条件差に伴う構造の整合性**を保ちやすく、装置・倍率・材料が切り替わっても表現の安定性を維持することが期待される。加えて、ViT は前述の**自己教師あり学習（DINO／DINOv2、MAE 等）との親和性が高く**、ラベル不要で得た汎用表現を**未学習の工程・材料領域へ転移**させやすい。以上より、**SSL 事前学習済み ViT を重み固定で用いる**ことが、案件ごとの再学習を要さずに**工程・材料・撮像条件をまたいだ汎化**を実現する方法論的根拠であり、本研究の中核を成す。

**工程横断ゼロショット欠陥検出「Promptable Patch Retrieval」**

---

## 3. アプローチ概要（重み固定・推論時適応）

### 3.1 基本方針
- **DINO／MAE／汎用 ViT（事前学習済み）を特徴抽出器として固定**する（重み更新は行わない）。
- 超高解像度画像をパッチ分割し、大量の**パッチ特徴量**を生成する。
- 判定は教師あり分類ではなく、主として以下に基づく。
  - **正常パッチ分布からの逸脱（anomaly score）**
  - **ROI 注釈に基づくプロトタイプ記憶（許容／不許容）**
  - **自然言語を構造化した適用条件（工程・材料・装置・期限等）**

### 3.2 推論フロー（概略）
1. **特徴抽出（固定 ViT）** — 高解像度画像をタイル化・パッチ化し、パッチ特徴を得る。
2. **一次検出（異常スコア化）** — 正常分布からの逸脱により、欠陥候補ヒートマップと ROI 候補を得る。
3. **人間フィードバック** — ROI（欠陥／非欠陥、許容／不許容）と自然言語コメント（適用範囲・例外条件）。
4. **構造化（LLM による JSON 化）** — 運用スキーマ（判定、適用範囲、優先度、有効期限など）へ変換する。
5. **補正レイヤ（Promptable Patch Retrieval）** — 欠陥候補 ROI の埋め込み（roi_embedding）とストア内のプロトタイプ（vit_embedding）を近傍照合し、類似度閾値および JSON の適用条件を満たすときに異常スコアを再構成する。
6. **最終判定** — NG、許容（False Positive の抑制）、要確認などを返す。

### 3.3 異常スコアの実装レンジ（参考）
一次検出では、パッチ特徴に対する **Mahalanobis 距離**、**k 近傍距離**（コアセットによる PatchCore 系）、**MAE のピクセル再構成誤差**などを組み合わせ、ヒートマップおよび ROI の切り出しに用いる。装置別・チャネル別に正常分布が乖離する場合は、特徴量メモリをメタデータ（工程・材料・装置タグ）で分割し、適用対象と突き合わせる。

## 4. NVIDIA DGX Spark が必要な理由
**超高解像度 × 工程横断 × マルチスケール**では、推論のみでも GPU スループットが律速となる。主因はモデルサイズより**データ側のスケール**にある：**1 枚あたり数千〜数万パッチ**、マルチスケールの繰り返し、および**大規模特徴量ストア**の保持とアクセス。DGX Spark の**大容量統一メモリ（128GB LPDDR5x）**と Tensor Core の性能は、PC 級 GPU における VRAM 枯渇とホスト転送を緩和する。LLM／VLM による補助推論を加える場合も、実験の回転に有利となる。

### 4.1 補足：ViT 規模との関係
「ViT が大きいため GPU が必要」とだけ述べるのは適切ではない。**ViT-Small は ResNet-50 と同程度のパラメータ**であり、モデルサイズだけが律速要因とは限らない。一方で DINO／DINOv2 や MAE の実用チェックポイントは **ViT-Large／Giant が多く**、モデル側のワークセットも増える。対外説明では**パッチ総量・マルチスケール・ストア規模を主な根拠**とし、Giant の採用は補強理由として述べるのが妥当である。

## 5. 研究開発新規性（論文化しやすい主張）
- **重み更新なし**でも工程・材料差をまたぎ、**特徴量メモリ**の設計指針を示す。
- ROI 注釈と自然言語を**推論時の条件**として統合し、誤判定を体系的に抑制する。
- **特徴抽出器の比較**（例：ImageNet 教師あり CNN と DINOv2 ViT、DINO のみと DINO＋MAE）および**同一パイプラインにおける工程横断での劣化曲線・HITL による回復量**を定量化できる。
- **ROI のみ／言語のみ／両方**の比較や、スコアの再重み付けとラベル上書きの差を検証可能とする。

---

## 6. 成果物（配布可能な構成）
- **固定モデル**：DINO／MAE／汎用 ViT（特徴抽出が目的）。必要に応じ C-RADIOv2 のような産業横断チェックポイントも評価候補とする。
- **推論パイプライン**：タイル化／パッチ化、特徴抽出、近傍探索（kNN／FAISS 等）、異常スコア・ヒートマップ、補正レイヤ。
- **UI／運用プロトコル**：ROI とコメント、コメントから JSON への変換スキーマと監査ログ。
- **特徴量ストア**：vit_embedding、annotation メタデータ、構造化 JSON、および適用メタ情報（後から検索できる索引設計）。

---

## 7. 評価計画
- **観点**：画像・領域単位の検出性能、False Positive／False Negative、工程・材料横断での汎化、フィードバック 1 件あたりの改善。
- **指標**：AUROC、AUPRC、FPR@TPR、領域 IoU／PRO／AUPRO 等。**Ground truth が乏しい製造データ**では、(1) MVTec-AD／VisA などの教師なし異常検知ベンチマークで手法の妥当性を確認し、(2) 実機ログではオペレータによる補正回数・誤適用率・レビュー待ちキューの滞留など運用 KPI を併記する二段構成とする。
- **堅牢性の示し方**：同一パイプラインで特徴抽出器を替えたときの AUROC／AUPRO。段階的ドメインシフト（同一工程→類似→異材料→異装置→異撮像）における劣化曲線。ドリフト検知後に **N 件の HITL プロトタイプ追加で性能がどこまで回復するか**を追う。

---

## 8. 想定リスクと対策（モデル・撮像・ドメイン）
- **事前学習ドメインと実機のギャップ** → 必要に応じ MAE のドメイン向け事前学習や、産業向けの強い表現モデルの検討。
- **撮像条件の変動が欠陥より支配的** → scope 単位での分布推定／再較正や、複数プロトタイプによる代表づけを行う。

---

## 9. まとめ
固定 ViT 系表現（**自己教師ありで得たセマンティックな埋め込みと、任意で MAE による再構成経路**）に基づき、ラベルのない状態から開始できる欠陥検出パイプラインを核とする。そのうえで**特徴量ストア**、**適用 scope を持つ構造化フィードバック**、**HITL によるプロトタイプの蓄積**を重ね、モデル再学習より運用側でドリフトへ適応できる。CNN と ImageNet 教師あり特徴と比較しても、自己教師あり事前学習済み ViT が工程・材料横断で有利になりやすいことを、公開データと製造側 KPI の両輪で示す。**NVIDIA DGX Spark** は、膨大なパッチとストアを前提とした評価の回転と、将来的なマルチモーダル補強のために活用する。
